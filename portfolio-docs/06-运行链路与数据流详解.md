# 运行链路与数据流详解

本文用于说明当前项目在一次真实任务中是如何运行的，重点回答四个问题：

1. 用户输入之后，系统按什么顺序运行。
2. 每一步会产生什么数据。
3. 这些数据如何在不同阶段之间传递。
4. Deep Research、长文分段写作、TN 可选生成是如何插入主流程的。

本文内容基于当前代码实现编写，编码为 `UTF-8`。

## 1. 先说清楚这不是传统后端架构

当前项目的“后台运行”本质上是：

- 前端页面负责收集输入、维护状态、驱动流程。
- 浏览器中的服务层直接调用 Gemini API。
- 本地状态对象负责在各阶段之间传递数据。

也就是说，当前并没有单独的业务后端服务。真正承担“编排器”角色的是前端中的这些文件：

- `App.tsx`
- `types.ts`
- `services/geminiService.ts`
- `services/promptAssets.ts`

因此，用户看到的是一个前端界面，但实际运行时，页面会连续触发多轮模型调用，并把中间产物逐步塞进一个统一的任务数据对象。

## 2. 整体运行总览

一次完整任务的主链路如下：

```text
用户输入主题与任务配置
-> 读取上传资料
-> 启动研究阶段
-> 综合研究
-> 量化研究
-> 人文研究
-> （可选）Deep Research
-> 合并为研究底稿 context
-> 展示研究底稿与来源 sources
-> 生成讨论方向 directions
-> 用户选择 selectedDirection
-> 生成文章大纲 outline
-> 用户审阅大纲
-> 提炼 writingInsights
-> 整理 evidenceCards
-> 规划 chunkPlan
-> 按 chunk 分段写作
-> 合并 draft
-> critique 审稿
-> rewrite 生成终稿 articleContent
-> （可选）生成 teachingNotes
-> 进入结果页与后续精修
```

## 3. 核心状态对象

### 3.1 `WritingTaskOptions`

这是任务配置对象，定义一次写作任务的约束条件。

当前字段包括：

| 字段 | 含义 |
| --- | --- |
| `genre` | 文体，例如商业分析、趋势解读 |
| `style` | 风格，例如理性克制、媒体型 |
| `audience` | 目标受众 |
| `articleGoal` | 文章目标 |
| `desiredLength` | 目标总字数 |
| `chunkLength` | 单轮写作长度，默认约 1500 字 |
| `includeTeachingNotes` | 是否生成 TN / 讨论指南 |
| `enableDeepResearch` | 是否叠加 Deep Research |

这个对象最早来自首页表单，之后会被反复传给研究、大纲、写作、TN 各个阶段。

### 3.2 `UploadedFile`

上传资料会被转成统一结构：

| 字段 | 含义 |
| --- | --- |
| `name` | 文件名 |
| `mimeType` | MIME 类型 |
| `data` | 文本内容或 Base64 数据 |
| `isText` | 是否为纯文本 |

处理规则是：

- `PDF` 会转成 Base64，作为 `inlineData` 传给模型。
- `TXT / MD / CSV` 会直接读成文本。
- 文本资料会优先进入研究提示词。

### 3.3 `GenerationState`

这是流程状态对象，用来驱动进度条、加载提示和页面切换。

字段包括：

| 字段 | 含义 |
| --- | --- |
| `step` | 当前所处阶段 |
| `progress` | 进度数值 |
| `message` | 当前阶段说明 |

当前阶段枚举为：

- `IDLE`
- `RESEARCHING`
- `REVIEWING_RESEARCH`
- `SELECTING_DIRECTION`
- `REVIEWING_OUTLINE`
- `WRITING`
- `COMPLETED`
- `ERROR`

### 3.4 `WritingProjectData`

这是整条链路中最重要的任务主对象。所有中间产物最终都会汇总到这里。

当前字段包括：

| 字段 | 含义 |
| --- | --- |
| `topic` | 主题，最终可能被终稿标题覆盖 |
| `sources` | 研究阶段整理出的来源列表 |
| `context` | 研究底稿 Markdown |
| `directions` | 讨论方向数组 |
| `selectedDirection` | 用户选中的讨论方向 |
| `outline` | 审阅通过的大纲 |
| `writingInsights` | 本次写作方法摘要 |
| `evidenceCards` | 证据卡 |
| `chunkPlan` | 长文分段写作计划 |
| `critique` | 审稿结果 |
| `articleContent` | 终稿正文 |
| `teachingNotes` | 可选 TN / 讨论指南 |
| `options` | 本次任务配置 |

可以把它理解成“本次写作任务的运行内存”。

## 4. 用户点击“开始研究”之后发生了什么

### 4.1 表单输入阶段

用户在首页完成这些输入：

- 主题 `topic`
- 文体 / 风格 / 受众 / 字数
- 是否生成 TN
- 是否启用 Deep Research
- 上传补充资料

然后点击“开始研究”。

### 4.2 前端先做两件事

在 `handleStartResearch` 中，系统会先做：

1. 把 `genState` 切到 `RESEARCHING`
2. 用当前任务配置重建一个新的空 `projectData`

这一步的目的，是确保本次任务不会混入上一次任务的残留数据。

### 4.3 调用 `gatherInformation`

真正的研究流程在 `services/geminiService.ts` 的 `gatherInformation` 中。

它的输入是：

- `topic`
- `uploadedFiles`
- `options`
- `onStatus`

它的输出是：

- `context`
- `sources`

其中：

- `context` 是完整研究底稿，Markdown 格式。
- `sources` 是来源数组，供研究审阅页展示。

## 5. 研究阶段如何跑

### 5.1 标准三路研究

系统默认一定会跑三路：

1. 综合研究
2. 量化研究
3. 人文研究

这三路都复用同一个研究函数 `runResearchTrack`，但每路的 `focus` 不同。

#### 综合研究关注什么

- 事实基线
- 时间线
- 主要参与方
- 关键事件
- 争议点

#### 量化研究关注什么

- 财务数据
- 经营指标
- 市场份额
- 用户规模
- 增长变化
- 可量化的判断依据

#### 人文研究关注什么

- 人物动机
- 组织文化
- 舆论反应
- 情境细节
- 关键表述
- 叙事张力所需的软信息

### 5.2 三路研究如何调用模型

标准三路研究使用的是：

- `getSearchModel()`
- 默认值为 `gemini-3.1-flash-lite`

调用方式是：

- `models.generateContent`
- 开启 `googleSearch` 工具
- 如果有上传文件，则把上传文件一起塞进 `contents.parts`

也就是说，三路研究不是只靠公开网页，而是：

```text
网页搜索结果
+ 上传资料
+ 当前任务约束
= 每一路研究底稿
```

### 5.3 三路研究产出什么

每一路都会产出两类东西：

1. `content`
2. `sources`

其中：

- `content` 是这一维度的研究底稿文本
- `sources` 是这一维度通过 grounding 提取出的来源链接

## 6. Deep Research 是怎么插进来的

### 6.1 它不是替代三路，而是叠加第四层

如果 `enableDeepResearch = true`，系统会在三路研究之后再追加一层：

- `Deep Research`

这意味着最终研究底稿结构会是：

```markdown
# 研究底稿

## 综合研究
...

## 量化研究
...

## 人文研究
...

## Deep Research
...
```

### 6.2 Deep Research 使用什么能力

Deep Research 使用的是官方 `Interactions API`，不是普通的 `generateContent`。

关键配置是：

- `agent: deep-research-pro-preview-12-2025`
- `agent_config.type: deep-research`
- `background: true`
- `store: true`

并且额外开启：

- `googleSearch`
- `urlContext`

### 6.3 Deep Research 的输入是什么

Deep Research 的输入不是从零开始，而是建立在三路研究已经完成的基础上。

它会拿到：

- `topic`
- `options`
- 三路研究合并后的 `baselineContext`
- 纯文本上传资料摘要

因此它做的是“补强式深挖”，不是重新跑一遍浅层研究。

### 6.4 Deep Research 为什么没有总超时

当前实现中没有设置总超时上限。

实现方式是：

1. 先 `interactions.create`
2. 然后每 `15` 秒轮询一次 `interactions.get`
3. 直到状态从 `in_progress / requires_action` 变成终态

终态包括：

- `completed`
- `failed`
- `cancelled`
- `incomplete`

### 6.5 Deep Research 如何持续反馈状态

轮询期间，系统会持续通过 `onStatus` 把运行信息推回前端。

显示内容类似：

- `正在启动 Deep Research Pro Preview...`
- `Deep Research 运行中（已耗时 2分15秒）: 正在后台检索与归纳...`
- `Deep Research 运行中（已耗时 4分30秒）: 正在执行中间步骤...`
- `Deep Research 已完成（总耗时 6分45秒），正在整理结果...`

这意味着即使耗时较长，前端也不会一直没有反馈。

### 6.6 Deep Research 产出什么

Deep Research 最终会产出：

1. `content`
2. `sources`

其中：

- `content` 会被写进研究底稿的 `## Deep Research` 章节
- `sources` 会和三路研究的来源一起合并去重

## 7. 研究阶段结束后如何进入下一步

当 `gatherInformation` 返回后：

- `projectData.context = context`
- `projectData.sources = sources`
- `projectData.options = taskOptions`
- `projectData.topic = topic`

然后 `genState.step` 变为 `REVIEWING_RESEARCH`。

这时界面会进入研究审阅页，用户可以看到：

- 完整研究底稿
- 来源列表
- 是否启用了 Deep Research 的提示

## 8. 讨论方向是怎么生成的

当用户确认研究底稿后，会触发 `generateDiscussionDirections`。

它的输入是：

- `topic`
- `context`
- `options`

它的目标是从研究底稿中生成 `5` 个可选讨论方向，输出是一个字符串数组 `directions`。

这里的本质是：

```text
研究底稿 context
-> 抽取可成立的几个“写法切口”
-> 返回 directions[]
```

如果用户不满意，还可以调用 `refineDiscussionDirections` 再重生一轮方向。

## 9. 大纲是怎么生成的

当用户选定一个方向后，会触发 `generateArticleOutline`。

它的输入是：

- `topic`
- `context`
- `selectedDirection`
- `options`

输出是：

- `outline`

当前大纲要求中已经明确约束了这些内容：

- 标题建议
- 导语策略
- 一级标题
- 二级展开点
- 每一部分的论证任务
- 总字数约束

如果用户给出反馈，还可以带着旧版 `currentOutline` 再生成一次。

## 10. 批准大纲之后，真正的写作流水线才开始

写作阶段不再是一口气直接出全文，而是进入 `generateArticlePackage`。

这是整个项目最核心的生产流水线。

输入是：

- `topic`
- `context`
- `direction`
- `outline`
- `options`

输出是一个完整产物包：

- `writingInsights`
- `evidenceCards`
- `chunkPlan`
- `critique`
- `articleContent`
- `teachingNotes`

## 11. 写作阶段内部的子流程

### 11.1 先生成 `writingInsights`

这是本次写作的短提示文件，作用是把研究底稿和大纲压缩成高价值写法说明。

它会提炼：

- 本次文章任务画像
- 最重要的写法原则
- 标题和开头建议
- 中段论证动作
- 必须避免的问题

后续写作阶段会把它直接喂回模型。

### 11.2 再生成 `evidenceCards`

这是把研究底稿结构化成“证据卡”。

它强调的不是泛泛总结，而是：

- 哪条事实最值得写
- 哪个数据最能支撑判断
- 哪个案例适合放在什么段落
- 哪个争议适合放在什么位置

它是后续成文时的“材料编排层”。

### 11.3 再生成 `chunkPlan`

这是长文稳定性的关键。

系统会根据：

- 目标总字数 `desiredLength`
- 单轮长度 `chunkLength`
- 当前大纲

把整篇文章拆成多个 `chunk`。

每个 `chunk` 至少包含：

- `index`
- `title`
- `sections`
- `targetLength`
- `purpose`

也就是说，长文不是“直接让模型写 3000 字”，而是“先规划，再按段推进”。

### 11.4 再按 chunk 逐轮写作

系统会循环跑 `generateChunk`。

每一轮都会拿到：

- 当前主题
- 当前方向
- 全文大纲
- `writingInsights`
- `evidenceCards`
- 本轮 `chunk`
- 前文 `previousText`

这里最重要的一点是：

- 第二轮以后，模型会拿到前文摘要和前文正文
- 同时被明确要求不要重复标题、开头和前文论点

这就是当前长文续写稳定性的核心实现方式。

### 11.5 合并成 `draft`

所有 chunk 写完之后，会被顺序拼接成初稿：

- `draft = chunks.join('\n\n')`

这个 `draft` 还不是最终交付版本，它只是完成了正文主体。

## 12. 初稿之后如何审查和改写

### 12.1 先跑 `critiqueArticle`

这一步会输出 `critique`，内容包括：

- 总体判断
- 结构问题
- 证据与论证问题
- 语言与 AI 腔问题
- 标题与开头建议
- 最优先修改清单

### 12.2 再叠加本地风格检查

除了模型审稿，系统还会跑一层本地规则检查 `lintStyle`。

它会扫描：

- AI 套话
- 夸大判断词
- 装饰性符号
- 普通词乱加引号

然后把结果附加到 `critique` 后面。

也就是说，当前审稿结果是：

```text
模型审稿 critiqueBase
+ 本地风格规则 lintStyle
= critique
```

### 12.3 最后跑 `rewriteArticle`

最终改写时，模型会同时拿到：

- `writingInsights`
- `evidenceCards`
- `outline`
- `critique`
- `draft`

然后生成最终正文 `articleContent`。

这一步的目标不是“重新写一篇”，而是：

- 保留有效信息
- 消除重复
- 降低 AI 腔
- 修正文风问题
- 提升标题、开头、结尾的完成度

## 13. TN / 讨论指南是如何生成的

TN 现在是可选产物，不是默认强制生成。

只有当：

- `includeTeachingNotes = true`

系统才会继续调用 `generateTeachingNotes`。

它的输入是：

- `topic`
- `direction`
- `articleContent`
- `options`

输出是：

- `teachingNotes`

因此 TN 的依赖关系是：

```text
终稿 articleContent
-> TN / 讨论指南
```

而不是和正文并行生成。

## 14. 最终哪些数据会进入结果页

当写作流水线完成后，系统会把这些字段塞回 `projectData`：

- `topic`
- `writingInsights`
- `evidenceCards`
- `chunkPlan`
- `critique`
- `articleContent`
- `teachingNotes`

然后把 `genState.step` 切到 `COMPLETED`，进入结果页。

所以结果页看到的不是单一正文，而是整套产物包。

## 15. 运行中真正被反复传递的“关键数据链”

如果只抓主干，当前项目的数据传递链可以简化成下面这条：

```text
topic + options + uploadedFiles
-> context + sources
-> directions
-> selectedDirection
-> outline
-> writingInsights + evidenceCards + chunkPlan
-> draft
-> critique
-> articleContent
-> teachingNotes（可选）
```

这是整个系统最核心的一条数据生命线。

## 16. Prompt 资产是怎么桥接进去的

项目当前并不是把所有 prompt 硬编码在单个 TS 大字符串里，而是从 Markdown 资产中读取长期规则。

当前加载的核心 prompt 资产有：

- `core_writing_skills.md`
- `universal_prompt.md`
- `ai_writing_workflow.md`
- `task_brief_template.md`

它们通过 `services/promptAssets.ts` 以 `?raw` 方式读入，然后被拼进每一轮系统指令。

因此，模型并不是“裸调用”，而是每一步都带着统一的长期规则运行。

## 17. 模型路由是怎么分工的

当前模型分工是：

### 17.1 搜索研究模型

- 存在于 `localStorage.SEARCH_MODEL`
- 默认值：`gemini-3.1-flash-lite`

负责：

- 综合研究
- 量化研究
- 人文研究

### 17.2 核心写作模型

- 存在于 `localStorage.GEN_MODEL`
- 默认值：`gemini-3.1-pro`

负责：

- 讨论方向
- 大纲
- writingInsights
- evidenceCards
- chunkPlan
- 分段写作
- 审稿
- 终稿改写
- TN 生成

### 17.3 Deep Research

不走普通模型路由，而是单独走：

- `agent: deep-research-pro-preview-12-2025`

它是附加研究层，不承担正文写作。

## 18. 重试、失败和中断是怎么处理的

### 18.1 通用重试

绝大多数模型调用都包裹在 `callWithRetry` 中。

当前策略是：

- 最多重试 `3` 次
- 退避基线 `1500ms`

### 18.2 错误传播

只要某一步抛错，`App.tsx` 就会把：

- `genState.step` 切到 `ERROR`
- `genState.message` 设置为对应阶段的失败说明

然后界面进入错误页。

### 18.3 Deep Research 的特殊点

Deep Research 不做总时长截断，但如果最终状态不是 `completed`，会直接报错。

这保证了：

- 慢可以慢
- 但不会无限静默
- 一旦进入失败态，前端可以收到明确错误

## 19. 一次真实任务的“数据生成顺序”

如果把一次完整任务按数据出生顺序列出来，大致如下：

1. `topic`
2. `taskOptions`
3. `uploadedFiles`
4. `general.content`
5. `quant.content`
6. `human.content`
7. `deepResearch.content` 可选
8. `context`
9. `sources`
10. `directions`
11. `selectedDirection`
12. `outline`
13. `writingInsights`
14. `evidenceCards`
15. `chunkPlan`
16. `chunk_1`
17. `chunk_2 ... chunk_n`
18. `draft`
19. `critique`
20. `articleContent`
21. `teachingNotes` 可选

这就是当前项目完整的一次生产链。

## 20. 一句话总结

当前系统不是“输入主题 -> 直接出文章”，而是一个分层生产流水线：

```text
研究分层
-> 方向选择
-> 结构审阅
-> 证据整理
-> 分段写作
-> 审稿回炉
-> 终稿输出
-> 可选 TN
```

这也是它相比一次性直出的核心价值所在。
