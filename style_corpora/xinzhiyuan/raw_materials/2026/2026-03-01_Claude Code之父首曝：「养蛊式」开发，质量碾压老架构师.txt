拒绝单线程：Git Worktrees的降维打击
在传统的AI辅助编程中，开发者50%的时间在等待AI响应或代码运行结果。
Boris看不得这种蠢办法，他用两个字回应
：
并行。
Boris团队的解法简单粗暴：直接拉3-5
个
Git Worktree
，每个Worktree运行一个独立的Claude Session。
这不比你搞一堆Repo副本香？
详细代码：https://code.claude.com/docs/en/common-workflows
#run
-parallel-claude-code-sessions-with-git-worktrees
Worktrees允许你在同一个仓库下拥有多个工作目录，所有分支共享同一个
.git
文件夹。
你能在同一份仓库历史下拉出多个工作目录，切任务不需要把分支、改动、聊天记录搅在一起；
每个目录都能绑定一条Claude会话，上下文更干净，改动也更容易回收
这样一来，你再也不用守着屏幕等测试结果，而是一个对话跑耗时测试，一个修bug，另一个写新功能，你只负责调度和验收。
这才是生产力暴涨的真实原因。
从「代码工人」到「架构师」：PlanMode的左右互搏
简单小脚本让Claude写当然没问题，但真正容易翻车的是复杂任务。
需求一多、依赖一杂、边界一模糊，AI就会开始走偏。
Boris
将整个流程优化之后，发现了一个技巧：复杂任务，先进
Plan Mode（计划模式）
。
Claude Code 启动界面：Plan Mode已开启（Shift+Tab循环切换）。Boris团队铁律——复杂任务先在这里倾注精力规划，再 1-shot 执行。
换一个更生动的比喻就是——养蛊，让
AI
查AI的岗。
第一条Claude负责出计划、做实现；第二条Claude刻意用更干净的上下文扮演Reviewer，专门盯边界条件、错误处理、风格一致性和可维护性。
审查意见再丢回去让第一条改。流程变成了「先计划，再执行；先实现，再复核」，生成的内容质量明显提升。
哪怕你是菜鸟开发者，也能通过这种「双机博弈」得到大师级的架构设计。
但是！这种方法千万不能滥用，盲目养蛊只会增加「代码熵」。
一旦跑偏，立刻按暂停，回退到Plan Mode把问题重新定义一遍再继续。
Mastering Claude Code：Boris Cherny团队精选11个高速度工程workflow。从Plan Mode起步，到Staff Engineer Reviewer博弈，再到persistent CLAUDE.md记忆，全流程可视化。
资产化记忆：CLAUDE.md就是外挂大脑
Boris有条铁律：一个坑你已经解释过一次，第二次还要解释，就该让流程帮你记住，而不是靠你忍耐。
要用好CLAUDE.md
文件，别让AI的智商随着Session结束就清零。
这是项目的「施工手册」，你希望它遵守的代码风格、目录约定、常用命令、不要碰的边界、你踩过的坑，都可以写进去。
纠正过一次的错误别只留在聊天记录里，直接让它把经验补进CLAUDE.md。
下次开新会话，能减少很多重复沟通。
一种常见的做法是：每次完成一轮开发后专门跑一遍「技术债清扫」，让AI把可以顺手解决的小问题列出来，再按优先级处理掉。
终结沟通内耗：MCP协议下的「直连秒杀」
还在Slack收到Bug，去CI查日志，再回IDE改代码？
更省事儿的方法，是把上下文打通。
MCP可以让你少当人肉搬运工。讨论在Slack里，你不用把一堆信息复制出来再解释一遍；日志在CI里，你也不用截屏拼图。
你可以让它去读相关上下文、调用对应工具，把问题复盘清楚，再把能落地的改动带回来。
这才是真正的爽点：省下注意力，而不是几次ctrl+c，ctrl+v。
Boris的方法，核心就一句话：别把自己耗在微观管理上。
把任务拆清楚，把并行跑起来，把经验沉淀下来这才是你该做的。
慢慢的你就能感觉到，你在带着AI把活干完。
CLAUDE.md写好，worktree开起来，Plan Mode用起来，AI就不会每次都像刚入职的新人那样从零开始。
