## 1. 系统架构总览

AI Writer 采用前后端分离架构：

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 UI | React + TypeScript + Vite | 交互界面与工作流状态管理 |
| 核心服务 | TypeScript（浏览器端） | 工作流引擎，串联所有写作阶段 |
| 后端服务 | Node.js HTTP Server | 提供资产加载与文件读取 API |
| 提示词资产 | Markdown 文件 | 可热加载的模块化提示词体系 |
| 风格配置 | JavaScript | 多风格库配置与切换 |

### 核心工作流（共 15 个阶段）

```
用户输入 → 资产加载 → 信息研究(3轨道+DeepResearch)
  → 讨论方向 → 大纲 → 参考模板匹配
  → Writing Insights → Evidence Cards → Chunk Plan
  → 分段写作 → 终稿缝合 → 终审循环(最多3轮)
  → 去AI化(诊断+修订+复检) → 句级终修 → TN生成
```

### 涉及的模型

| 用途 | 模型 |
|------|------|
| 研究/搜索 | `gemini-3.1-flash-lite` |
| 生成/写作 | `gemini-3.1-pro-preview` |
| Deep Research | `deep-research-pro-preview` |

## 2. 启动与初始化

### 2.1 后端服务

后端是一个纯 Node.js HTTP 服务器，监听 `127.0.0.1:8787`，提供以下核心 API：

| 路由 | 功能 |
|------|------|
| `/api/content/prompt-assets` | 加载运行时提示词资产 |
| `/api/content/style-profiles` | 返回可用风格库列表 |
| `/api/reference-templates/catalog` | 返回参考模板文章目录 |
| `/api/reference-templates/full-text` | 返回指定文章全文 |

### 2.2 多风格库支持

系统内置多个风格库，每个风格库拥有独立的人格定义、提示词资产、参考文章语料与反模式规则。用户可在前端自由切换，系统会自动加载对应风格的全部资产。

### 2.3 前端静态资产

前端在构建时通过 Vite 打包以下核心 Markdown 资产：

| 资产 | 用途 |
|------|------|
| 核心写作总纲 | 全局写作方法论与质量标准 |
| 通用写作提示 | 写作时的通用约束规则 |
| AI写作工作流 | 工作流阶段说明文档 |
| 任务卡模板 | 写作任务的结构化格式 |

## 3. 阶段一：用户输入与任务配置

用户通过前端界面配置写作任务，主要参数包括：

| 参数 | 说明 | 示例 |
|------|------|------|
| 风格库 | 选择写作风格体系 | 复旦商业文章、晚点 LatePost |
| 文体 | 文章类型 | 案例分析、趋势解读 |
| 风格 | 写作调性 | 理性分析、洞察型 |
| 目标受众 | 读者画像 | 企业管理者 |
| 文章目标 | 内容诉求 | 解释趋势并给出判断 |
| 目标字数 | 期望篇幅 | 3000-8000字 |
| 写作主题 | 核心话题 | AI 医疗行业的发展趋势 |

此外，用户还可上传补充资料文件，并选择是否启用 Deep Research 和 TN 教学指南生成。

系统将所有参数组装为一份**结构化任务简报（Task Brief）**，贯穿后续所有阶段作为上下文锚点。

---

## 4. 阶段二：运行时资产加载

每次写作任务开始时，系统通过后端 API 动态加载一组 Markdown 格式的运行时提示词资产，包括：

| 资产类型 | 说明 |
|----------|------|
| Master Persona | 主人格定义——明确系统的身份定位、行为纪律、任务理解规范与输出质量门槛 |
| 反AI文风规则 | 用于约束生成文本的文风，避免典型 AI 生成痕迹 |
| 去AI化护栏与检查维度 | 多维度的去AI化诊断与修订规则体系 |
| 风格库反模式 | 当前风格库需要规避的写作模式 |
| 子人格（风格库特有） | 部分风格库包含按文体自动匹配的子人格 |

这些资产以模块化方式组织，可独立更新和扩展，无需修改代码。

## 5. 阶段三：信息研究（Research）

### 核心机制

系统自动执行 **3条并行研究轨道**，利用 Gemini Google Search 工具实时搜索互联网信息：

| 轨道 | 搜索重点 |
|------|----------|
| 综合研究 | 事件背景、时间线、关键主体、争议点与基础事实 |
| 量化研究 | 财务数据、业务指标、市场份额、投融资、估值 |
| 人文研究 | 人物履历、公开表态、组织变化、舆论反应、媒体叙事 |

每条轨道独立调用搜索模型，以 Markdown 研究笔记格式输出结果。对无法确认的信息标注"待核实"，对搜索结果冲突处单独标出。

### Deep Research（可选）

用户可开启 Deep Research 模式，系统将额外调用 Google Deep Research Agent 进行深度调研，输出直接保留原生结果，不做二次总结。

### 结果合并

所有轨道的研究结果合并为统一的**主题资料库（Ammo Library）**，供后续所有写作阶段调用。

---

## 6. 阶段四：讨论方向生成

基于主题资料库，系统自动生成 **5个差异化的讨论方向**，每个方向需包含明确判断、切入角度和展开潜力。

用户可以：
- 直接选择一个方向
- 给出"补充偏好"，系统据此重新生成方向

此阶段确保文章不是泛泛而谈，而是有明确的论证立场。

---

## 7. 阶段五：文章大纲生成

基于资料库、用户选定的讨论方向与参考模板文，系统生成一份结构化大纲，要求：

- 有明确标题、开头任务、中段推进、尾段收束
- 每个大段标注论证任务
- 参考模板文的结构与节奏，但不照搬
- 支持用户反馈修订，可多轮迭代

---

## 8. 阶段六：参考模板匹配与装载

### 匹配算法

系统从参考文章库中自动匹配最适合本次任务的 **3篇参考模板文章**，算法分 4 步：

1. **风格纯度过滤**：排除低价值、软文、活动通知等文章，按加权评分计算风格纯度
2. **任务相关度打分**：综合词汇匹配、标题命中、质量分、参考价值等维度
3. **Boost 加权**：对文体/风格/受众匹配的文章给予额外加分
4. **多样性选择**：使用 MMR（最大边际相关性）算法，确保选出的文章互不雷同

### 全文装载

选定的文章从后端加载全文，连同其结构模式、开头方式、收束方式等元信息一并注入后续 Prompt。

---

## 9. 阶段七：写作方法提炼（Writing Insights）

系统生成一份精炼的写作方法指南，包括：

- 本次任务画像分析
- 模板文借鉴要点
- 核心写作规则（5条）
- 需要规避的文风风险

控制在 800-1200 字，作为后续写作阶段的方法论参考。

---

## 10. 阶段八：证据卡整理（Evidence Cards）

将资料库中与当前讨论方向直接相关的信息，整理为可快速调用的**证据卡片**：

- 关键事实与数据
- 企业案例
- 可引用的专家观点
- 争议点与待核实信息

分组存储，方便写作时精准引用。

---

## 11. 阶段九：分段写作规划（Chunk Plan）

系统将大纲拆分为多个写作 **Chunk**，每个 Chunk 包含：

| 字段 | 说明 |
|------|------|
| title | 本段标题 |
| sections | 负责的大纲小节 |
| targetLength | 目标字数 |
| purpose | 本段写作任务描述 |

Chunk 数量根据目标字数和单轮写作长度自动计算，也可根据大纲标题数量推断。

---

## 12. 阶段十：分段正文写作（Chunk Writing）

按 Chunk Plan 逐段写作正文。核心特性：

- **上下文连续**：每个 Chunk 可以看到前文，确保衔接连贯
- **资料锚定**：写作时引用证据卡和资料库中的真实数据
- **模板对齐**：参考模板文的节奏、密度和气质
- **反AI约束**：全程执行反 AI 文风规则，避免套话和装饰性修辞

---

## 13. 阶段十一：终稿结构缝合（Assembly）

当文章由多个 Chunk 拼接而成时，自动触发缝合阶段。重点处理：

- Chunk 接缝处的重复开头/收束
- 转场突兀与段落断裂
- 信息堆叠与重复论证
- 子标题补全（3-6 个 `##` 标题）

原则：**最小必要修订**——能删一句解决的问题，不整段重写。

---

## 14. 阶段十二：杂志级终审循环（Editorial Review）

模拟杂志终审流程的 **审校-修订循环**（最多 3 轮）：

```
for 每一轮:
  1. 终审诊断 → 输出结构化审校报告
  2. 若 ready=yes 或无阻塞问题 → 退出循环
  3. 否则 → 根据诊断策略执行修订
```

### 诊断策略

| 策略 | 适用场景 |
|------|----------|
| `structure_tune` | 子标题缺失、结构断裂、接缝问题 |
| `continuity_tune` | 段落承接、收束、重复或推进不顺 |
| `micro_polish` | 句级用词、节奏、AI 腔残留 |

每轮最多返回 6 个问题，采用渐进式审校——前轮已解决的问题不重复提出。

---

## 15. 阶段十三：商业稿去AI化（Commercial Humanizer）

独立于终审的专项清理流程，包含 **3个子步骤**：

1. **初检**：诊断残余 AI 痕迹，输出结构化报告
2. **修订**：执行最小必要修改（保留标题、结构、论证顺序）
3. **复检**：再次诊断，确认清理效果

### 检查维度

| 维度 | 检查内容 |
|------|----------|
| 内容模式 | 夸大意义、知名度堆砌、宣传语言、公式化段落 |
| 语言模式 | AI 高频词、否定式排比、三段式罗列、虚假范围 |
| 版式模式 | 破折号/粗体过多、emoji、标题列表残留 |
| 对话痕迹 | 协作式套话、谄媚语气、通用积极结论 |

核心原则：**去AI化 ≠ 口语化**——保持商业杂志的克制、客观与分析型语体。

---

## 16. 阶段十四：句级终修（Final Polish）

发稿前最后一位 line editor，只做句级和短语级的最小必要修改：

- 对照参考模板文收紧语气、节奏与克制程度
- 清理残余 AI 腔、假转折、空泛抽象词
- 补全缺失的子标题（如有）
- 不重新打开前几轮已解决的问题

---

## 17. 阶段十五：TN 教学指南生成

根据最终成文生成可选的教学笔记/讨论指南，内容包括：

- 适用教学场景
- 核心讨论问题
- 可用板书结构
- 课堂风险提醒

---

## 18. 辅助功能

### 18.1 Copilot 对话

用户完成写作后，可通过 Copilot 与编辑对话。系统自动识别用户意图——如果是修改请求，则转化为 refine 指令。

### 18.2 选区精修

用户选中一段文字后给出修改指令，系统仅修改选中部分，不影响其余内容。

### 18.3 整稿精修

用户给出全局编辑要求，系统在保留整体结构的前提下修订全文。

---

## 19. 系统指令构建机制

每个阶段的 System Instruction 由 `buildSystemInstruction()` 函数统一构建，采用模块化拼装结构：

```
{本阶段角色描述}
{Master Persona}
{子人格（如适用）}
{风格库反模式（如适用）}
{反AI文风硬约束}
```

当运行时资产加载失败时，系统使用硬编码的兜底人格，确保工作流不中断。

部分风格库支持**子人格自动选择**——根据用户选定的文体/风格/目标关键词，自动匹配最合适的子人格。

---

## 20. 本地风格检查（Lint Style）

在终审和去AI化阶段，系统先用**正则表达式**对文稿做本地预检，检测 20+ 种常见问题：

| 类别 | 示例 |
|------|------|
| AI 连接句 | "换句话说""更重要的是""说到底" |
| 商业黑话 | "赋能""闭环""抓手""范式迁移" |
| 模糊主体 | "有人认为""业内普遍认为" |
| 过度工整句 | "不是……而是……""既……又……" |
| 聊天腔/协作痕迹 | "你会发现""希望这对你有帮助" |
| 格式问题 | 破折号过多、粗体过多、emoji |

检查结果以 Markdown 格式注入终审和去AI化的提示词中，辅助模型定位问题。
